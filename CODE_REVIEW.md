### Блок D — AniList і глобальний стан

Таблиця кроків (орієнтир коду):

| Крок | Що зробити |
|------|------------|
| **D.1** | Маппінг anime-api ↔ AniList, `ourAnimeId → anilistId`. |
| **D.2** | Автопрогрес + синх з AniList. |
| **D.3** | Wish list / статуси, UI + бекенд. |
| **D.4** | Глобальний стан auth/AniList (підхід обрати пізніше). |

Блок D — **окрема гілка** після стабілізації A/B; реалізація **поетапно**, без спроби зробити весь «Overview як на AniList» в одному релізі.

#### Узгоджена ціль (продукт + техніка)

- **Підключення профілю:** користувач **прив’язує** свій AniList через **OAuth**; на бекенді зберігається зв’язок `ourUser ↔ anilistUser` і **токени** (не тримати секрети на клієнті).
- **Синхрон прогресу:** після перегляду епізоду на сайті — **оновлення прогресу в AniList** (GraphQL mutations). Нумерація епізодів на кшталт «як у всіх» допомагає, але API все одно вимагає **`mediaId` (AniList)**, тому без шару **маппінгу** `ourAnimeId → anilistId` не обійтися.
- **Відображення у своєму профілі:** дані на кшталт **Activity History**, **Genre Overview**, агрегати (total anime, days/episodes watched), стрічка активності — **витягувати з AniList (GraphQL)** і показувати у власному UI; на **v1** допустимо «схоже за змістом», а не піксель-в-піксель як на anilist.co.
- **Джерело правди (узгодити на старті):** для першої версії логічно: **істина на сайті при перегляді тут → пуш на AniList**; повний двосторонній синк і конфлікти (дивився тут vs оновив у застосунку AniList) — пізніший етап.

#### Етапи впровадження (план обговорення)

*(Літери в колонці «Етап» нижче — лише **підетапи блоку D**, не плутати з **A.1** / **A.2** у розділі «Готові».)*

| Етап | Зміст |
|------|--------|
| **A. Підключення** | OAuth AniList, callback на бекенді, збереження токенів; у UI — «Підключено / Відв’язати». |
| **B. Маппінг** | Для тайтлів, які дивляться на сайті, мати `anilistId` (ручне / пошук через API / пізніше автодоповнення). |
| **C. Пуш прогресу** | Подія «переглянуто епізод» → сервер оновлює список/прогрес на AniList; обробка помилок, rate limit; опційно черга повторів. |
| **D. Вітрина в профілі** | Спочатку прості метрики → жанри → складніші віджети (heatmap, activity); залежить від того, що віддає API одним або кількома запитами. |
| **E. Стан на клієнті** | Дані AniList краще кешувати через **RTK Query** (або TanStack Query), якщо підключите — окремий кеш для серверних даних; **Redux** — для сесії/UI та прапорців («синхронізується», помилки), без дублювання всього GraphQL у великий slice «вручну». |

#### Ризики та обмеження

- **Ліміти GraphQL** — не дьоргати API на кожному рендері; кеш, оновлення за дією користувача.
- **Токени** — лише сервер; на клієнті тільки статус підключення.
- **Не все з веб-UI AniList** гарантовано відтворюється одним запитом — закласти ітерації UI.
